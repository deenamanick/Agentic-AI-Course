# Beginner Teaching Guide

## Who this course serves

Assume every class includes mixed experience:

- Project managers and product owners
- DevOps, cloud, and operations engineers
- UX and visual designers
- Developers from different languages
- Teachers, homemakers, students, and career-switchers

Never use previous coding experience as the measure of intelligence or potential.

## The teaching loop

Use the same loop for every concept:

1. **Relate:** Start with an everyday analogy.
2. **Show:** Demonstrate the visible result.
3. **Name:** Introduce only the technical words needed.
4. **Build:** Copy and run the smallest example.
5. **Change:** Ask learners to modify one safe value.
6. **Break:** Cause one expected error.
7. **Explain:** Let learners describe the flow in their own words.
8. **Reflect:** Connect the concept to different job roles.

## Three learning levels

### Understand — required for everyone

The learner can explain the purpose and data flow without code.

### Practice — normal classroom target

The learner can copy, run, change, and test a guided example.

### Challenge — optional

The learner can extend, refactor, or troubleshoot independently.

Never let optional challenges block progress to the next concept.

## How to explain code

- Explain the output before the syntax.
- Read code in small blocks, not whole files.
- Use “this line asks Python to…” instead of jargon-first descriptions.
- Introduce no more than five new terms in one session.
- Keep a visible glossary.
- Allow learners to annotate code in their own language.

## Inclusive classroom activities

- Use roleplay for request flows.
- Pair a technical learner with a domain learner, switching who explains.
- Ask UX learners to identify loading and error states.
- Ask project learners to map stages and acceptance criteria.
- Ask operations learners to identify services, ports, and failures.
- Ask nontechnical learners to test clarity and identify unexplained words.

## Facilitator questions

Prefer:

- “What do you expect to happen next?”
- “Which part checked the input?”
- “How would you explain this to a customer?”
- “What is one safe thing we can change?”
- “What information would help us debug this?”

Avoid:

- “This is basic.”
- “You should already know this.”
- “It is obvious.”
- “Just copy it.”

## A simple 90-minute session

| Time | Activity |
|---|---|
| 0–10 min | Analogy and roleplay |
| 10–20 min | Instructor demonstration |
| 20–35 min | Explain five or fewer new terms |
| 35–55 min | Guided copy-and-run practical |
| 55–65 min | Break |
| 65–75 min | Change one value and rerun |
| 75–83 min | Safe failure and troubleshooting |
| 83–90 min | Learner explanation and checklist |

## Assessment

Assess understanding in several ways:

- Explain a diagram
- Arrange flow cards in order
- Identify request and response JSON
- Demonstrate a working command
- Change one feature
- Diagnose a prepared error

Typing speed and memorized syntax should not dominate the score.
